# 🌐 浏览器 NAT 穿透检测原理（小白与硬核双拼指南）

在过去，想要准确检测自己的网络到底是哪种 NAT（比如所谓的“全锥形”还是“对称型”），通常需要下载专门的软件（比如各种网络测试工具或者游戏加速器客户端）。

但在本项目中，我们**仅仅依靠一个网页和一行 Python 脚本**，就做到了传统客户端才能做的事。下面是一份既适合小白理解、又包含专业硬核技术的原理解析。

---

## 🐣 第一部分：通俗科普（小白请看这）

### 1. 什么是 NAT？它在干什么？
你可以把 NAT（网络地址转换）想象成你们**小区的门卫大爷**。
- 你的电脑在内网（小区内部），有个内部房号（如 `192.168.1.100`）。
- 快递员（外部互联网）只知道小区的**公共地址**（公网 IP）。
- 当你想跟外面联系时，门卫大爷会在传达室开一个**临时小窗口（Port 端口）**，帮你把信件递出去，并记住“这个窗口是给 100 房用的”。

### 2. 为什么需要穿透（P2P）？
当你用网页跟朋友一对一视频、或者打联机游戏时，如果直接让你们的两台电脑互相发数据（P2P），速度最快且不卡。但这需要你的“门卫大爷”允许别人的信件从外面直接投递到你的小窗口里。

### 3. 四种大爷（NAT 的类型）
- 🚪 **全锥形 (Full Cone)**：最宽松的大爷。只要你开过一次窗口，**任何人**从外面往这个窗口塞信，大爷都给你送进来。P2P 联机体验极佳（网络类型 A）。
- 🛂 **地址受限 (Address-Restricted)**：大爷有点警惕。如果你之前给张三写过信，那么**只有张三**（认 IP）能往窗口塞信，李四的信会被丢掉。
- 🔒 **端口受限 (Port-Restricted)**：大爷非常严格。你给张三的手机发过短信，那不仅必须是张三，还必须是**张三用手机**回信才行（认 IP 还认原本的 Port），否则扔掉（网络类型 B）。
- 🎭 **对称型 (Symmetric)**：彻底变态的大爷。你每联系一个不同的人，大爷就强行给你换一个**全新的小窗口**。外界根本猜不到你的窗口号，P2P 打洞几乎不可能（网络类型 C/D，必须用中继服务器兜底）。

## 💻 第二部分：黑魔法原理（硬核技术全解析）

在桌面 App 里，你可以直接开一个 UDP Socket 发包来测试大爷的性格。但是**在浏览器（网页）里，为了安全，UDP 原生套接字是被彻底封杀的！**
我们唯一能用的后门，就是 **WebRTC 的 ICE（互动连接建立）机制**。

本项目通过 **双节点服务器 (Twin-Server)** 加上对 WebRTC 协议协议层的“合法欺骗”完成了整套检测：

### 核心步骤 1：探测映射规则 (Mapping)
*目标：区分是对称型（变态大爷），还是锥形（固定窗口）。*

**我们的做法：**
在 JS 也就是前端代码中，我们创建一个 `RTCPeerConnection`，并强行给它同时塞入三个不同的 STUN 服务器：
1. 主节点 IP_A，端口 3478
2. 主节点 IP_A，端口 3479 （同 IP，不同端口）
3. 辅助节点 IP_B，端口 3478 （完全不同的 IP）

WebRTC 引擎会乖乖地从**同一个本地端口**，向这三个目标发送 STUN 询问包，并拿到自己在外网的映射端口（`srflx` Candidate）。
- 如果浏览器收集到：打给 A 是端口 `8001`，打给 B 也是 `8001` -> 端口不变，这就是 **锥形 NAT**。
- 如果打给 A 的 3478 是 `8001`，打 A 的 3479 变成了 `8002`，打 B 是 `8003` -> 端口疯狂变幻，这就是 **对称型 NAT (Symmetric)**。立刻结束测试。

---

### 核心步骤 2：探测过滤规则 (Filtering) —— 也就是区分 Full Cone
*目标：在确认了外网端口是 `8001` 的前提下，试探其他没联系过的陌生 IP 能不能通过这个端口把包送进来。*

**网页的硬伤：**
由于浏览器的限制，如果这时候用辅助服务器 B 直接朝客户端的 `8001` 送 UDP 包，浏览器引擎在底层就会把这个包当成垃圾直接丢掉！网页 JS 根本连包的影子都看不到！

**绝妙的解决方案：欺骗 WebRTC 状态机**
1. **生成假票据**：我们利用 JS，把生成好的本地 WebRTC Offer SDP 拿出来，强行对其**魔改 (Munge)**。我们凭空捏造一个假的后端机器人，给它编造一套 `ice-ufrag` 和 `ice-pwd` 账号密码，并写入一个假的 `a=candidate`（比如指向 `192.0.2.1`），然后作为 Answer 塞回给引擎的 `setRemoteDescription`。
2. **激活监听 (Checking state)**：这会导致浏览器信以为真，进入 ICE Checking 状态。此时，**浏览器会在 `8001` 端口进入高度警戒状态**，准备接收来自对方的打洞包（STUN Binding Request）。
3. **后门伪造发包**：此时前端通过 Fetch 请求调用主服务器说：“我已经准备好了，账号密码已发送，我的公网端口是 8001”。
4. **服务器代发 (Active Probe)**：主服务器立刻通知毫无关联的 **辅助服务器 B**。利用 Python 后端，我们严格根据 RFC 5389 规范，自己手动“徒手”捏造一个 STUN 打洞数据包！
   - 我们把 `USERNAME` 拼装成 `浏览器ufrag : 服务器ufrag`
   - 我们用浏览器的 `password` 进行严谨的 **HMAC-SHA1 哈希加密**（`MESSAGE-INTEGRITY` 属性）。
   - 我们算出整个二进制包的 **CRC32 校验码**（`FINGERPRINT` 属性）。
   - 最终由辅助服务器 B 发射这个 UDP 导弹至客户端的 `8001` 端口！
5. **判定结果**：
   如果这时候是在 **Full Cone** 环境下，大爷放行了这个陌生包裹。浏览器的底层 C++ WebRTC 引擎收到这个包，解密发现：**“握草，密码对上了，原来是自己人！”**。随后由于 ICE 规范中的 **Peer Reflexive (prflx)** 机制，浏览器会自动向服务器 B 乖乖地回传一个 UDP 回执（STUN Binding Success Response）。
   
   只要我们的服务器 B 截获到了这个从用户家里发出的 UDP 回执 —— **破案了，完美的全锥形（Full Cone）！** 

如果服务器 B 没收到，主服务器再换个临时端口去发，如果收到了就是地址受限，都没收到就是经典的端口受限。

## 🎯 总结

这套检测器没有写一行底层的 C++ 或浏览器插件扩展，纯粹利用了 **HTML5 原生的 WebRTC 协议缺陷与容错机制**，在服务端徒手构造包含加密指纹的协议包反向敲门，最终在网页端实现了以前需要复杂桌面客户端才能完成的**高级防火墙穿透行为分析**。
